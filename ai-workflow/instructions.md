# Instructions — 内容生成规则

本文件是共识文档生成的内容规范。所有 /new-consensus 和 /update-consensus 的输出都必须遵守。

## 语言规范

### 禁用技术术语

用业务语言替代技术术语：

| ❌ 禁用 | ✅ 使用 |
|--------|--------|
| DTO / VO / BO | 数据结构 / 数据对象 |
| Controller | 接口处理 / 对外接口 |
| Service | 业务逻辑 |
| Repository / DAO | 数据访问 |
| Entity | 业务实体 |
| Endpoint | 接口地址 |
| Payload | 请求内容 / 响应内容 |

### 禁用模糊表述

| ❌ 禁用 | ✅ 改为 |
|--------|--------|
| "等" | 必须穷举所有项 |
| "类似的" | 必须具体说明 |
| "相关的" | 必须明确关系 |
| "等等" | 必须完整列出 |
| "一些" | 必须给出具体数量或范围 |

### 格式规范

- 所有列表必须有明确编号（1. 2. 3. 或 - 缩进层级）
- Mermaid 流程图节点名称使用中文业务语言
- 代码块必须标注语言类型
- 表格必须有表头

## 9 项共识文档的内容要求

### 1. 资料完整性确认
- 分三类标注：✅ 已获取 / ⚠️ 不完整 / ❌ 缺失
- 缺失项必须标注"对第 {n} 项有影响"

### 2. 业务流程图
- 每个主流程一张图
- 每个流程图后有文字说明
- 异常分支用虚线或不同颜色区分

### 3. 功能模块划分
- 每个模块的"职责"和"边界"分开写
- 模块之间的依赖关系用文字或图明确

### 4. 核心数据结构
- 实体之间的关系（一对一 / 一对多 / 多对多）用文字明确
- 每个字段的业务含义必须说明

### 5. 前后端契约
- API 使用 RESTful 风格
- 状态码约定采用 HTTP 标准
- 业务错误码统一前缀（如 E001、E002）

### 6. 分工边界
- 灰色地带的归属必须明确（例：表单校验由前端负责，后端做二次校验）

### 7. 风险与疑问
- 每个问题编号（Q1、Q2...），方便后续追踪
- 技术风险附带初步应对方案

### 8. 前瞻性确认
- 至少覆盖 2-3 个可能的扩展方向
- 评估当前设计的改造成本（高/中/低）

### 9. Lovable 差异
- 对比至少一个同类产品的实现
- 差异点对技术实现的特殊要求必须说明

## Commit Message 格式

```
ame: {action} v{YYYY-MM-DD}{, extra info}
```

例：
- `ame: consensus v2026-04-15`
- `ame: consensus update v2026-04-20, merged #12 #18`
- `rules: fix E3+B1 (38→43)`
