# /review

结构化代码校验，逐项输出通过/未通过。

## 参数

- `feature`（必填）：要 review 的功能或变更范围
- `role`（必填）：`backend` 或 `frontend`

## 执行流程

### 1. 加载上下文
按 role 加载对应 knowledge（与 /impl 相同的分层策略）+ 设计文档 + 共识文档 + 变更代码。

### 2. 结构化校验

#### 通用检查
| # | 项目 | 说明 |
|---|------|------|
| G1 | 命名规范 | 类、方法、变量是否符合约定 |
| G2 | 无魔法值 | 常量是否定义为命名常量或枚举 |
| G3 | 异常处理 | 无吞异常，有日志或抛出 |
| G4 | 设计对齐 | 实现与设计一致 |

#### 后端专项
| # | 项目 | 说明 |
|---|------|------|
| B1 | API 路径 | 与共识文档契约一致 |
| B2 | 请求/响应结构 | 字段、类型与契约一致 |
| B3 | DDD 分层 | 业务逻辑在正确层（不在 Controller） |
| B4 | 数据库命名 | snake_case |
| B5 | 参数校验 | 对外接口校验完整 |
| B6 | 框架工具 | 使用框架工具而非自实现 |

#### 前端专项
| # | 项目 | 说明 |
|---|------|------|
| F1 | 组件库 | 使用指定组件库 |
| F2 | API 封装 | 通过统一 request 模块 |
| F3 | 状态管理 | 遵循项目方案 |
| F4 | 接口对齐 | 路径和参数与契约一致 |

### 3. 输出报告

```markdown
## Review 报告 — {feature}

### 总结
- 通过：{n}
- 未通过：{n}
- 需讨论：{n}

### 未通过详情

#### {编号} {项目名}
- 文件：`{path}:{line}`
- 问题：{描述}
- 建议：{方向}

### Knowledge 更新建议
- {如有}
```

### 4. 人工决策

提示：
> Review 完成。
> - 全部通过 → 进入测试
> - 未通过 → 建议执行 /impl 修复后重新 /review

### 5. 修复循环

用户选择修复：
1. 未通过项作为修复指令传给 /impl
2. /impl 完成后重新 /review
3. 重复直到全部通过
