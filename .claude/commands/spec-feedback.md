# /spec-feedback

发现设计缺口或冲突时，生成结构化反馈。

## 参数

- `feature`（必填）：涉及的功能
- `issue`（必填）：问题简述（或从当前对话上下文提取）

## 执行流程

### 1. 生成结构化反馈

```markdown
# Spec Feedback — {feature}

**日期**：{YYYY-MM-DD}
**发现者**：{dev}
**严重性**：高 / 中 / 低

## 问题描述
{具体是什么问题}

## 触发场景
{在什么情况下发现的——/impl 中编码时？/review 中校验时？}

## 影响范围
- 影响哪些模块
- 影响哪些角色（后端/前端/测试）
- 是否阻塞当前迭代

## 冲突类型
- [ ] 设计文档内部矛盾
- [ ] 设计与共识文档不一致
- [ ] 共识文档有缺失
- [ ] 共识文档有歧义
- [ ] 新需求超出共识范围

## 建议方案

### 方案 A：更新设计文档
- 修改内容：...
- 影响范围：仅本项目
- 预估成本：低

### 方案 B：升级到共识更新
- 修改内容：...
- 影响范围：可能影响其他项目
- 预估成本：中（需走 /update-consensus 流程）

## 建议处理路径
{基于上述分析，推荐哪个方案及理由}
```

### 2. 保存反馈

保存到 `docs/feedback/{date}-{feature}-{issue-slug}.md`。

### 3. 团队决策

提示：
> 反馈已生成。建议团队讨论后决定：
> - 更新设计 → 由设计负责人修改 docs/design/
> - 升级到共识更新 → 去 ai-workflow 仓库执行 /update-consensus
> - 标为已知问题 → 在 .claude/knowledge/ 中记录
>
> 在决策明确前，当前实现暂停或使用临时方案。
```

## 与反馈汇总的联动

这些反馈文件会在 P5 阶段被汇总：
- 由指定的规则维护者定期 review
- 批量转为 GitHub Issue 影响共识文档
- 触发 ai-workflow 仓库的 /update-consensus

形成"项目端发现 → 规则端修正 → 所有项目受益"的闭环。
